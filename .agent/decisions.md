## 2026-07-15 · Day 6

### 决策 019：前后端时间参数统一规范

- **决策**：前后端时间传参统一使用毫秒级时间戳（`long`），Java `LocalDateTime` → `Instant.toEpochMilli()`，前端 `dayjs().valueOf()`
- **理由**：Day 5 时间范围选择器联调暴露了格式不统一问题（ISO 8601 vs 毫秒戳 vs `yyyy-MM-dd HH:mm:ss`），导致跨天/跨月边界数据查询异常
- **影响**：Controller 层参数类型统一为 `Long`；前端 api 封装层统一 `dayjs` 毫秒戳入参；规避时区与格式转换问题

### 决策 020：ML 环境一键初始化脚本

- **决策**：编写 `ml/setup.sh`（76 行），5 步完成环境全流程：创建 Python venv → 生成模拟数据 → 导入 MySQL → 特征工程 → 训练 LSTM + Prophet
- **理由**：新成员手动搭 ML 环境需 2-3 小时（Python 依赖、数据生成、特征提取、模型训练），一键脚本压到 ~10 分钟
- **影响**：`ml/setup.sh` 成为环境初始化的唯一入口；readme 和 CLAUDE.md 均引用此脚本；训练流程可复现

### 决策 021：TorchScript 静态图适配方案

- **决策**：固定 LSTM 输入序列长度（`seq_length=168`），对输入数据做 padding 对齐，适配 TorchScript 静态图约束
- **理由**：TorchScript 导出要求固定输入 shape，而原始 PyTorch 模型支持动态长度 → 导出后 Flask 加载报 shape mismatch
- **影响**：`train_lstm.py` 增加序列长度参数；Flask `app.py` 推理前对输入自动 padding/truncate；模型元数据 `lstm_meta.pt` 记录 `seq_length`

## 2026-07-20 · 接口边界与文档同步

### 决策 033：模型版本查询与本地同步分离

- **决策**：`GET /api/v1/model/versions` 保持纯查询；`POST /api/v1/model/versions/sync` 负责扫描并登记本地模型产物。
- **理由**：查询接口应保持无副作用，避免列表刷新时隐式同步或清理模型文件。
- **影响**：模型同步由运维人员显式触发，模型页面显示数据库发布版本与 Flask 实际运行版本的生效状态。

### 决策 034：告警研判查询只读缓存

- **决策**：`GET /api/v1/alert/events/{id}/judgement` 只返回已有研判缓存；没有缓存时返回空结果或提示重新研判。只有显式 `POST` 才调用 LLM。
- **理由**：避免普通详情查询隐式产生 LLM 费用和延迟，并让重新研判动作可追踪。
- **影响**：前端需要在无缓存时引导调度员点击“重新研判”。

### 决策 035：红色告警统一为待确认工单草稿

- **决策**：全局文案使用“待确认工单草稿”，不再使用“自动建单”或“自动提交工单”。
- **理由**：红色告警会触发强提醒和草稿预填，但正式工单必须由调度员确认提交。
- **影响**：避免用户误以为系统已绕过人工确认权直接派发工单。

### 决策 036：后续重训练入口收敛为 LSTM

- **决策**：产品后续只使用 LSTM 重训练；Prophet 后端兼容能力、历史版本和旧模型文件暂时保留，待确认无依赖后移除页面重训练入口。
- **理由**：减少模型管理入口歧义，同时不破坏历史预测记录和已有兼容逻辑。
- **影响**：当前文档不宣称 Prophet 已完全删除，后续移除页面按钮时再同步更新代码和文档。

### 决策 037：文档同步基准
