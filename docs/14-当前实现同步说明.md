## 2. 已完成的核心功能

### 2.1 大屏与数据

- 历史负荷、恢复段、实时模拟负荷分段展示。
- 恢复段与实时段之间使用前端平滑 bridge 曲线连接。
- bridge 点不写入数据库，不影响后续真实整点数据归档。
- 页面展示数据来源，区分历史数据、恢复模拟数据和实时模拟数据。
- 实时负荷当前仍是演示用模拟数据，不代表真实 SCADA 数据。

### 2.2 Agent 聊天

- 支持 SSE 流式事件：`thinking`、`text`、`chart`、`done`、`error`。
- 助手 Markdown 文本以完整消息发送，前端在流结束前后都能正确渲染。
- ECharts option 保存到 `conversation.chart_option`，历史会话重新打开可恢复图表。
- `done` 事件包含数据来源、时间范围、模型版本和模拟数据标识等可信度信息。

### 2.3 告警与工单

- 告警文案使用固定模板，保证低延迟；工单详情的 AI 智能研判真实调用 LLM，失败时降级为规则研判。
- 告警生命周期已支持恢复时间、确认时间和确认人。
- 已读与人工确认处置分开记录。
- 红色告警只生成“待确认工单草稿”，不会自动提交正式工单。
- 调度员确认后才创建正式 `alert_ticket`。
- 未创建工单的告警不显示处置时间线，也不会复用其他工单的 actions。
- 工单进入 `RESOLVED` 后，调度员关闭前必须提交完整结构化反馈；`resolution` 仍保留自由文本摘要，不替代反馈。
- `GET/PUT /api/v1/tickets/{id}/feedback` 按 `ticket_id` 一对一幂等保存反馈。处理人身份取当前认证用户，复核人和复核时间由关闭流程写入；已关闭工单普通角色只读，系统管理员修正进入审计链路。
- 历史已关闭工单没有反馈时返回 `LEGACY_MISSING`，不会批量伪造分类。告警反馈聚合只统计 `CLOSED` 工单，准确率按 `trueAlertCount / (trueAlertCount + falsePositiveCount) * 100` 计算，分母为 0 返回 `null`。
- Agent 的 `query_alert_ticket` 仅返回反馈摘要和脱敏告警证据，不具备反馈写入、关闭或正式建单能力。
- 告警研判接口：
  - `GET /api/v1/alert/events/{id}/judgement`：只读已有缓存，没有缓存时返回空。
  - `POST /api/v1/alert/events/{id}/judgement`：显式重新调用研判。

#### P0-05 告警运行状态、证据与送达确认

- V18 新增 `alert_runtime_state`、`alert_delivery_metric`，并扩展 `alert_event` 的 occurrence、规则/证据快照和来源时间字段。
- 运行态以 `nodeId:ruleId:type` 唯一 `stateKey` 管理。阈值和拓扑模拟告警均在同一持久化状态机下经历 `NORMAL -> PENDING -> ACTIVE -> RECOVERED`；同一来源实例的重复或旧序列不会再次判断。应用重启后的下一条有效实时点直接从数据库状态继续，不会重新推送历史事件或自动建正式工单。
- `AlertEvent` 是历史 occurrence，运行态、已读、人工确认、恢复、工单解决/关闭及 WebSocket delivery-ack 是独立语义。恢复不会覆盖已读、人工确认或工单状态；红色告警仍只生成待确认草稿。
- 告警触发时保存规则与证据快照。普通列表只返回结构化摘要字段；完整 JSON 保留给受控的管理/审计用途。拓扑告警明确标记为 `topologySimulated=true`，不表示真实潮流或 N-1 校验结果。
- 浏览器在 WebSocket 告警写入状态并完成一个渲染帧后调用 `delivery-ack`，失败最多重试一次。确认不改变业务状态。端到端延迟为源观测时间至服务端收到确认时间，P50/P95 采用 nearest-rank；无样本显示“暂无呈现确认样本”。
- 当前仅保证单实例的数据库持久化和重启恢复，不包含 P1 事件级 shelving 审批、交接班、外部消息总线或多实例高可用。

### 2.4 模型管理与重训练

- 模型版本列表是纯查询，不会隐式同步或清理本地模型产物。
- `POST /api/v1/model/versions/sync` 专门负责同步本地模型产物。
- 模型页面展示数据库发布版本、Flask 实际运行模型及“已生效/待重启/推理服务异常”状态。
- 重训练任务会记录数据时间范围、样本数、耗时、失败原因和产物清单。
- 启动重训练前，会从 MySQL 的 `load_data` 导出最新负荷和天气字段，覆盖生成 `featured_load_data.csv`，再调用训练脚本。
- LSTM 重训练会同步生成模型、输入 scaler、目标 scaler、特征列和元信息，并在登记前校验产物完整性。
- 产品已决定后续不再使用 Prophet。当前代码仍保留 Prophet 后端兼容能力和模型页入口，待下一步移除页面重训练按钮；历史 Prophet 版本和旧模型文件暂时保留，避免破坏历史数据。
- 版本切换或重训练完成后不会自动热重载 Flask，需要根据页面状态重启 Flask 服务。

### 2.5 预测运营与天气

- 支持预测质量摘要、线上预测复盘、滚动 MAPE/RMSE/MAE、峰值误差。
- 有验证集 RMSE 时生成运行参考区间，标注为参考区间，不宣称严格概率置信区间。
- 已接入未来天气预报缓存：`weather_forecast` 表、同步任务、查询接口和模型推理输入。
- 新训练的 LSTM 可使用未来 24 小时温度和湿度；旧模型仍可兼容，天气不可用时回退历史天气。

### 2.6 拓扑风险、节点预测与场景推演

- 新增 `grid_node`、`grid_edge` 拓扑表，内置“区域 → 变电站 → 馈线”三级演示拓扑。
- `load_data`、`prediction_result`、`alert_event` 增加 `node_id`；历史根区域数据仍归属 `REGION-DEMO`。
- `GridNodeSimulationService` 为 4 条馈线生成独立时序、节点噪声和周期性异常，再向上汇总生成变电站数据，来源标记为 `NODE_SIMULATION` / `NODE_SIMULATION_ANOMALY`。
- `PredictService` 支持 `forecast(nodeId)`；定时预测遍历所有在役节点分别调用 Flask，并按节点写入 `prediction_result`。原 `GET /api/v1/predict/forecast` 不传 `nodeId` 时仍预测根区域。
- `GET /api/v1/topology/risk` 使用节点实际模拟负荷和节点预测峰值计算风险；父节点按子节点聚合，父子风险标记 `alertRootNodeCode` 和 `alertDeduplicated`。
- `TopologyAlertScheduler` 只为去重后的根风险节点创建 `TOPOLOGY_RISK` 告警，写入 `root_event_id` 和 `impact_load_mw`，避免同一风险链路产生重复告警。
- 新增 `POST /api/v1/topology/scenario`，支持节点或线路故障影响推演，返回受影响节点、受影响负荷、可转供余量和预计未供负荷。
- 前端“拓扑风险”页面支持节点/线路故障推演；Agent 新增 `query_grid_scenario`，`query_grid_risk` 返回根告警和已合并节点。
- 拓扑计算、场景推演和根告警识别均由确定性规则完成；LLM 只负责基于结构化证据解释风险、归纳可能根因和生成处置建议。
- 当前所有节点数据和场景结果仍是模拟数据，不代表真实 SCADA/EMS、线路潮流、保护动作或 N-1 校验结果。

## 4. 当前边界与待办

- 实时负荷仍是演示模拟数据，尚未接入真实 SCADA 或生产数据源。
- Flask 当前按启动时加载模型，数据库发布版本与运行模型需要通过页面状态和服务重启确认一致。
- Prophet 仅作为历史兼容项，后续可在确认无历史依赖后清理训练脚本、产物和依赖。
- 训练任务当前已有数据库记录，但后台执行仍依赖本地进程，不是分布式任务队列。
- 预测参考区间是基于验证集 RMSE 的运行参考，不等同于严格概率预测区间。
- 拓扑尚未接入真实区域、变电站和馈线数据，场景推演仍不是潮流计算、保护动作分析或 N-1 校验。
- 节点预测当前复用同一个 Flask 模型分别推理，后续可按节点类型或负荷类别训练专用模型。
- 前端 Vitest、TypeScript 检查和生产构建已通过；Vite 仍提示主 bundle 较大。

## P0-04 实时遥测同步

- `V17__realtime_telemetry.sql` 建立独立秒级遥测表；`load_data` 仍只承载小时级历史/训练数据。
- 实时点使用 `observedAt`、`receivedAt`、`sourceInstanceId` 和实例内 `sequence`；以 `(dataSource, sourceInstanceId, sequence)` 幂等。
- `BAD`（空、NaN、Infinity 或负负荷）仅持久化审计，不进主曲线、WebSocket 或告警；`LATE` 与 `ESTIMATED` 可展示但不会更新告警输入。`STALE` 是当前流状态。
- 服务启动从最近 30 分钟持久化记录恢复内存快照，不回放 WebSocket、不重新触发告警；超过 24 小时的数据按批清理。写库失败时保留内存路径并标记 `PERSISTENCE_DEGRADED`，不伪造恢复。
- Dashboard 标示模拟来源、质量和新鲜度，REST 与 WebSocket 使用稳定键去重；旧消息按 `LEGACY_UNKNOWN` 兼容。

## P0-06 模型发布一致性

- LSTM 训练产物写入 `ml/models/<modelVersion>/` 不可变目录。训练先写入同级临时目录，写全文件与 `manifest.json` 后原子重命名。
- `artifactChecksum` 对 manifest 列出的模型文件按相对路径字典序计算；每项文本为 `relativePath + "\n" + fileSha256 + "\n"`，其 UTF-8 拼接文本再做 SHA-256，`manifest.json` 不参与自身校验。
- Flask 先在临时 `ModelBundle` 中完成目录限制、manifest、文件 checksum、反序列化和冒烟推理，全部通过才原子切换。失败时旧 bundle 持续预测。
- 数据库发布版本和 Flask 运行版本由 version/checksum 分别记录；只有完全一致才显示 `已生效`。旧平铺产物及历史 Prophet 为 `LEGACY_UNVERIFIED`，只读且不能宣称可靠回滚。

## P0-07 工单反馈与告警准确率

- V20 新增 `ticket_feedback`，与 `alert_ticket` 一对一，`actions_taken` 为 JSON 数组，结构化字段覆盖告警分类、根因、措施、影响负荷和处置效果。
- 新工单关闭前必须存在完整反馈；反馈校验、复核人写入、工单状态变更和 `CLOSE` 时间线位于同一事务边界。关闭失败不会产生 `CLOSED` 状态，重复关闭幂等返回。
- `ALERT` 反馈必须关联真实 `alert_event`；`PREWARNING` 可无 `alert_id`，明确标识为预测风险提前处置，不描述为已触发真实告警。
- 处理人和复核人均由后端认证身份写入，不能由前端传入。处理人必须是当前工单处理角色；已关闭反馈普通角色只读，系统管理员修正会记录审计。
- 查询反馈时返回真实告警的结构化证据摘要，保留历史 `ruleVersion`、数据来源和触发时间，不向普通用户暴露完整快照 JSON。Agent 只读反馈摘要，不具备反馈写入、关闭或正式建单工具。
- `GET /api/v1/tickets/feedback/metrics` 只统计 `CLOSED` 工单；`LEGACY_MISSING` 单独计数，准确率分母为 0 时返回 `null`。本阶段只沉淀运营数据，不自动修改告警规则或模型。
