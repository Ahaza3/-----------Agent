## 4. 机器学习方案

### 4.1 核心策略：训练与推理分离

```
┌─────────────────────────────────────────────────────────┐
│  训练阶段（离线，Python）                                  │
│  ┌──────────┐   ┌──────────┐   ┌──────────────────┐     │
│  │ 历史数据  │ → │ 特征工程  │ → │ PyTorch 训练 LSTM │     │
│  │ (CSV)    │   │ (pandas) │   │ Prophet 训练       │     │
│  └──────────┘   └──────────┘   └────────┬─────────┘     │
│                                         │ 导出模型       │
│                                    ┌────▼────────┐      │
│                                    │ model.pt     │      │
│                                    │ model.onnx   │      │
│                                    └─────────────┘      │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  推理阶段（在线，Java）                                    │
│  ┌──────────┐   ┌──────────────┐   ┌──────────────┐     │
│  │ 实时数据  │ → │ DJL 加载模型  │ → │ 预测结果      │     │
│  │ (API)    │   │ (PyTorch引擎) │   │ (JSON响应)    │     │
│  └──────────┘   └──────────────┘   └──────────────┘     │
└─────────────────────────────────────────────────────────┘
```

**为什么训练用 Python、推理用 Java？**
- 训练 LSTM 是 Python 生态的主场（PyTorch 教程丰富、调试方便）
- 模型训练一次即可，导出为 `model.pt` 或 ONNX 格式
- Java 端通过 DJL（Deep Java Library）加载模型进行推理，无缝融入 Spring Boot

### 4.2 模型训练方案（Python 脚本）

| 组件 | 技术 | 用途 |
|:-----|:-----|:-----|
| 深度学习 | PyTorch 2.x | LSTM 模型训练、验证、导出 |
| 历史兼容模型 | Prophet | 保留旧版本展示与兼容，不作为后续重训练入口 |
| 数据处理 | pandas + numpy | 特征工程、数据清洗 |
| 标准化 | scikit-learn StandardScaler | 导出均值 / 标准差供推理端预处理 |
| 模型格式 | TorchScript / ONNX | 跨语言模型交换格式 |

```

```

### 4.3 Java 端推理引擎：Python Flask 微服务（主力方案）

| 对比维度 | Flask 微服务 | DJL | 结论 |
|:---------|:------------|:----|:-----|
| 环境兼容性 | ✅ Python 装了就能跑 | ❌ 常有 `UnsatisfiedLinkError` | Flask 零环境问题 |
| 调试便利性 | ✅ `curl` 单独测试 | ❌ 嵌在 Java 里难定位 | 独立测试，出问题一眼定位 |
| 开发速度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 搭 Flask 服务约 30 分钟 |
| 16 天风险 | 🟢 低 | 🔴 高（可能浪费 2–3 天） | **Flask 是决定性选择** |

**选择理由**：16 天周期最怕环境问题卡进度。Flask 推理微服务约 60 行代码，Java 端 OkHttp 调用约 50 行。训练好的 PyTorch 模型加载到 Flask 进程中，Java 通过 HTTP 调用 `/predict/batch` 获取预测结果。

> **⚠️ Day 3 验证**：如果你有信心 DJL 在你的环境中能跑通，可以在 Flask 服务跑通后尝试 DJL 替换。但**半天搞不定立刻切回 Flask**，别死磕。

```python
# ml/app.py — Flask 推理服务（约 60 行）
from flask import Flask, request, jsonify
import torch, numpy as np

app = Flask(__name__)
model = torch.jit.load("lstm_model.pt")
model.eval()

@app.route("/predict/batch", methods=["POST"])
def predict_batch():
    features = np.array(request.json["features"], dtype=np.float32)
    tensor = torch.from_numpy(features).unsqueeze(0)
    with torch.no_grad():
        preds = model(tensor).squeeze().tolist()
    return jsonify({"predictions": preds})

@app.route("/health")
def health():
    return jsonify({"status": "ok"})

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000)
```

### 4.4 备选方案：DJL（Deep Java Library）

如果团队环境 DJL 能顺利跑通（事先验证！），可以用 DJL 直接加载模型，免去额外进程。但**务必 Day 3 第一天就验证**，不要等到 Day 6 才踩坑。

## 5. LLM / Agent 框架

### 5.1 LLM API 选型

| 对比维度 | DeepSeek | 通义千问 (Qwen) | OpenAI | 本地部署 |
|:---------|:---------|:---------------|:-------|:---------|

### 5.2 Agent 框架选型：自研轻量

| 对比维度 | LangChain4j | Spring AI | 自研轻量 | 结论 |
|:---------|:------------|:----------|:---------|:-----|
| 学习成本 | 中 | 中高 | 低 | 16 天周期自研最可控 |
| 灵活性 | 中 | 中 | ⭐⭐⭐⭐⭐ | Function Calling 流程简单 |
| 代码透明度 | 黑盒 | 黑盒 | 白盒 | 答辩解释友好 |
| 依赖体积 | 大 | 大 | 小 | Docker 镜像更轻 |
| Spring Boot 集成 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 无框架锁定 |

**最终选择**：**轻量自研**。Function Calling 的核心逻辑不到 200 行代码，不需要引入 LangChain4j 或 Spring AI 的复杂性。

### 5.3 自研 Agent 核心架构（Java）

```java
// Agent 核心编排伪代码
@Service
public class AgentCore {

    private final OkHttpClient httpClient;
    private final ToolRegistry toolRegistry;
    private final ObjectMapper objectMapper;

    /**
     * 处理用户自然语言查询
     * @param message    用户消息
     * @param history    多轮对话历史
     * @param emitter    SSE 流式输出
     */
    public void chat(String message, List<Message> history, SseEmitter emitter) {
        // 1. 构造 messages: system_prompt + tools_schema + history + user_msg
        List<Map<String, Object>> messages = buildMessages(message, history);

        // 2. 调用 LLM API（支持 Function Calling）
        ChatResponse response = callLLM(messages, toolRegistry.getDefinitions());

        // 3. 如果返回 function_call → 执行工具 → 追加结果 → 递归调用 LLM
        if (response.hasFunctionCall()) {
            String toolResult = executeTool(response.getFunctionCall());
            messages.add(buildToolResultMessage(toolResult));
            response = callLLM(messages, null);  // 二次调用
        }

        // 4. 流式返回结果
        emitter.send(response.getContent());
    }

    private String executeTool(FunctionCall call) {
        Tool tool = toolRegistry.get(call.getName());
        return tool.execute(call.getArguments());
    }
}
```
