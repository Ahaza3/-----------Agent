## 二、三层次范式定义（复述）

| 层级 | 模式 | 判定标准 | 合规要求 |
|:-----|:-----|:---------|:---------|
| 💡 **启发式** | 与 AI 讨论方案，拓展思路 | 人类做架构决策，AI 仅提供观点/对比 | 代码由人书写，AI 未参与生成 |
| ✍️ **精确式** | 手写骨架 → AI 辅助优化 | 人类先写核心结构/接口，AI 补全细节 | 骨架为人写，AI 仅做补全和优化 |
| 🔒 **约束式** | 前置规范 → AI 按规范生成 | 有明确接口契约/规范约束，AI 在笼子里生成 | 生成前有书面规范，生成后通过 Review |
| 🚫 **Vibe Coding** | 无约束直接生成 | 无前置规范，无人工骨架，直接让 AI "做一个XX功能" | **禁止** — 必须整改 |

## 四、疑似 Vibe Coding 详析

### 4.1 `Dashboard/index.tsx` — 疑似 Vibe Coding

**文件**：[frontend/src/pages/Dashboard/index.tsx](../frontend/src/pages/Dashboard/index.tsx)（317 行）

**AI 生成特征：**
- 代码结构高度完整：同时实现了快捷时间切换、自定义日期选择器、ECharts dataZoom（inside + slider）、预测叠加、markLine、统计卡片行——这种"一次到位"的完整度是 AI 典型特征
- 内联 `<style>` 标签中的 CSS 含有 `@media` 响应式断点，说明一次性生成了完整的自适应方案
- `fmtMw`、`quickToRange` 等工具函数与组件写在同一文件中（人类通常会抽取到 utils）

**问题：**
- 无明显前置规范文档约束——Dashboard 的具体交互细节（dataZoom 配置、预测叠加方式、颜色方案）未在接口设计文档或 UI 原型中预先定义
- 组件过长（317 行），包含图表配置逻辑 + 状态管理 + UI 渲染 + CSS，职责混杂
- 预测叠加逻辑（手动构建 forecastData 数组拼接实际值末点与预测值）较复杂，未经独立单元测试

**建议整改：**
1. 将图表配置逻辑抽取为 `useLoadChartOption` hook
2. 将 `fmtMw`、`quickToRange` 抽到 `utils/` 目录
3. 补充 Dashboard UI 原型或交互说明文档作为"前置规范"
4. 人工逐行审查 dataZoom 和预测叠加的边界条件（空数据、单点数据、预测失败）

### 4.2 `train_lstm.py` / `train_prophet.py` — 疑似 Vibe Coding

**文件：**
- [ml/train_lstm.py](../ml/train_lstm.py)（342 行）
- [ml/train_prophet.py](../ml/train_prophet.py)（128 行）

**AI 生成特征：**
- 两个文件都包含完整的"加载数据 → 预处理 → 训练 → 评估 → 导出"流水线
- `train_lstm.py` 包含 LSTM 模型定义、训练循环、早停机制、学习率调度——这些是经典的 "AI 生成的 ML 训练脚本" 模板
- 虽然有 `model_comparison.md` 记录了评估结果（MAPE 3.53% / 4.71%），但训练过程本身可能未经过人工逐步验证

**问题：**
- 训练脚本属于"一次性运行"代码，容易形成"AI 生成 → 跑通 → 不审查"的路径
- LSTM 模型架构（层数、hidden size、dropout）的选择理由未记录在 `.agent/decisions.md` 中
- 特征工程（19 个特征列）的选择理由未记录

**建议整改：**
1. 在 `.agent/decisions.md` 中补充决策记录：LSTM 架构选择的理由、特征选择的依据
2. 人工逐行审查 `train_lstm.py` 中的模型定义和训练循环
3. 补充 `model_comparison.md` 中关于"为什么 LSTM 优于 Prophet"的技术分析

> **附录**：AI 代码生成特征识别方法
>
> 本次审计使用的 AI 特征识别启发式：
> 1. **文档均匀性** — 每个方法都有结构完全一致的 Javadoc/TSDoc → AI 特征强
> 2. **完成度异常** — 组件一次性覆盖 loading/empty/error/normal 四种状态 → AI 特征中
> 3. **风格零差异** — 跨文件/跨语言的注释风格、命名风格完全一致 → AI 特征强
> 4. **人味缺失** — 无 TODO、FIXME、个人习惯写法、不完美但故意的代码 → AI 特征强
> 5. **复杂度集中** — 单个文件包含多种职责（如 Dashboard 的 UI+状态+图表+样式） → AI 特征中
