## 3. 核心表设计

### 3.1 `load_data` — 历史负荷数据表

**用途**：存储逐小时负荷数据，是预测模型的唯一数据源。所有数据查询、统计、大屏展示都基于此表。

```sql
CREATE TABLE load_data (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '自增主键',
    node_id     BIGINT COMMENT '归属拓扑节点 ID',
    time        DATETIME NOT NULL COMMENT '数据时间点（精确到小时）',
    load_mw     FLOAT NOT NULL COMMENT '负荷值(MW)',
    temperature FLOAT COMMENT '温度(°C)',
    humidity    FLOAT COMMENT '湿度(%)',
    is_holiday  TINYINT DEFAULT 0 COMMENT '是否节假日(0=否,1=是)',
    hour        TINYINT COMMENT '小时(0-23)',
    day_of_week TINYINT COMMENT '星期几(0=周一,6=周日)',
    month       TINYINT COMMENT '月份(1-12)',
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',

    UNIQUE INDEX uk_load_node_time (node_id, time),
    INDEX idx_load_node_time (node_id, time),
    INDEX idx_hour (hour),
    INDEX idx_day_of_week (day_of_week),
    INDEX idx_month (month)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='历史负荷数据表';
```

| 字段 | 类型 | 必填 | 说明 |
|:-----|:-----|:----:|:-----|
| id | BIGINT | ✅ | 自增主键 |
| time | DATETIME | ✅ | 数据时间点，精确到小时。UNIQUE 约束防止重复导入 |
| load_mw | FLOAT | ✅ | 负荷值，单位 MW。允许小数 |
| temperature | FLOAT | ❌ | 温度，可为空（无气象数据时） |
| humidity | FLOAT | ❌ | 湿度，可为空 |
| is_holiday | TINYINT | ✅ | 0=非节假日，1=节假日。默认 0 |
| hour | TINYINT | ✅ | 冗余字段，加速按小时聚合查询 |
| day_of_week | TINYINT | ✅ | 冗余字段，加速按星期聚合 |
| month | TINYINT | ✅ | 冗余字段，加速按月份聚合 |

> **设计说明**：`hour`、`day_of_week`、`month` 为冗余字段。虽然可从 `time` 计算得出，但预计算存储可显著加速统计聚合查询（如"计算每天 18:00 的平均负荷"）。

### 3.2 `prediction_result` — 预测结果表

```sql
CREATE TABLE prediction_result (
    id               BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '自增主键',
    node_id          BIGINT COMMENT '预测目标拓扑节点 ID',
    predict_time     DATETIME NOT NULL COMMENT '预测的时间点',
    predicted_load   FLOAT NOT NULL COMMENT '预测负荷值(MW)',
    lower_bound      FLOAT COMMENT '置信区间下界',
    upper_bound      FLOAT COMMENT '置信区间上界',
    model_version_id BIGINT COMMENT '关联模型版本ID',
    created_at       DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '预测生成时间',

    INDEX idx_predict_time (predict_time),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='预测结果表';
```

| 字段 | 类型 | 必填 | 说明 |
|:-----|:-----|:----:|:-----|
| predict_time | DATETIME | ✅ | 预测所针对的未来时间点 |
| predicted_load | FLOAT | ✅ | 预测的负荷值 |
| lower_bound | FLOAT | ❌ | 80% 置信区间下界（Prophet 输出） |
| upper_bound | FLOAT | ❌ | 80% 置信区间上界 |
| model_version_id | BIGINT | ❌ | 关联 model_version.id，追溯预测来源 |

> **存储策略**：每次模型推理生成 24h 预测 → INSERT 24 条。前端取 `created_at` 最大的 24 条即为最新预测。历史预测保留用于精度对比。

> **P2 运行说明**：当关联模型版本存在验证集 RMSE 时，`lower_bound` / `upper_bound` 按 `predicted_load ± 1.2816 × RMSE` 生成，作为 80% 参考区间展示；线上复盘通过 `predict_time` 与 `load_data.time` 对齐，计算实际 MAPE、RMSE、MAE 和峰值误差。

### 3.3 `alert_event` — 告警事件表

```sql
CREATE TABLE alert_event (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '自增主键',
    node_id         BIGINT COMMENT '告警对象拓扑节点 ID',
    root_event_id    BIGINT COMMENT '关联的根告警事件 ID',
    impact_load_mw   FLOAT COMMENT '预计影响负荷(MW)',
    trigger_time    DATETIME NOT NULL COMMENT '触发时间',
    level           VARCHAR(10) NOT NULL COMMENT '告警级别: RED/ORANGE/YELLOW',
    type            VARCHAR(20) NOT NULL DEFAULT 'THRESHOLD' COMMENT '告警类型: THRESHOLD/TREND/ANOMALY',
    current_value   FLOAT COMMENT '当前负荷值(MW)',
    threshold_value FLOAT COMMENT '触发阈值(MW)',
    rule_id         BIGINT COMMENT '关联告警规则ID',
    ai_analysis     TEXT COMMENT 'AI 分析文案（模板生成）',
    suggestion      TEXT COMMENT '建议措施',
    is_read         TINYINT DEFAULT 0 COMMENT '是否已读(0=未读,1=已读)',
    resolved_at     DATETIME COMMENT '告警解决时间',
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',

    INDEX idx_trigger_time (trigger_time),
    INDEX idx_level (level),
    INDEX idx_is_read (is_read),
    INDEX idx_type (type)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='告警事件表';
```

| 字段 | 类型 | 必填 | 说明 |
|:-----|:-----|:----:|:-----|
| level | VARCHAR(10) | ✅ | 仅允许 RED / ORANGE / YELLOW |
| type | VARCHAR(20) | ✅ | THRESHOLD（阈值告警）/ TREND（趋势预警）/ ANOMALY（异常检测） |
| ai_analysis | TEXT | ❌ | 🚫 禁飞区 NFZ-3 — AlertTemplate 手写模板生成 |
| suggestion | TEXT | ❌ | 建议措施，可选，P1 阶段补充 |
| resolved_at | DATETIME | ❌ | 告警解决时间，NULL 表示未解决 |

> **去重逻辑**：同一小时内同一 level 不重复插入（Service 层 `isDuplicate()` 方法检查）。

### 3.7 `grid_node` / `grid_edge` — 拓扑节点与连接关系（P1）

当前拓扑使用区域、变电站、馈线三级逻辑拓扑。运行时由节点模拟馈送器为馈线生成独立时序和异常，再通过拓扑层级汇总到变电站；`allocation_ratio` 仍仅用于模拟数据初始化和缺失数据回退，接入真实 SCADA/EMS 数据后应替换为节点实际测量值。

```sql
CREATE TABLE grid_node (
    id                BIGINT PRIMARY KEY AUTO_INCREMENT,
    node_code         VARCHAR(64) NOT NULL UNIQUE,
    node_name         VARCHAR(100) NOT NULL,
    node_type         VARCHAR(32) NOT NULL,
    parent_id         BIGINT,
    allocation_ratio  DECIMAL(8,5) NOT NULL DEFAULT 1.00000,
    rated_capacity_mw FLOAT,
    voltage_level     VARCHAR(32),
    status            VARCHAR(20) NOT NULL DEFAULT 'IN_SERVICE',
    topology_version  VARCHAR(32) NOT NULL DEFAULT 'demo-v1',
    sort_order        INT NOT NULL DEFAULT 0,
    created_at        DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at        DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_grid_node_parent (parent_id),
    INDEX idx_grid_node_type (node_type)
);

CREATE TABLE grid_edge (
    id               BIGINT PRIMARY KEY AUTO_INCREMENT,
    from_node_id     BIGINT NOT NULL,
    to_node_id       BIGINT NOT NULL,
    edge_type        VARCHAR(32) NOT NULL DEFAULT 'HIERARCHY',
    capacity_mw      FLOAT,
    status           VARCHAR(20) NOT NULL DEFAULT 'IN_SERVICE',
    topology_version VARCHAR(32) NOT NULL DEFAULT 'demo-v1',
    created_at       DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE INDEX uk_grid_edge (from_node_id, to_node_id, topology_version)
);
```

`load_data`、`prediction_result` 和 `alert_event` 通过 `node_id` 关联拓扑节点；历史单区域数据统一归属 `REGION-DEMO` 根节点。节点预测按 `node_id` 分批写入 `prediction_result`，拓扑风险父节点按子节点聚合。`root_event_id` 用于根告警标识，`impact_load_mw` 用于记录根告警影响负荷。当前结果来源标记为模拟数据，不代表真实潮流计算。

## 4. 索引策略

| 表 | 索引名 | 类型 | 索引列 | 查询场景 |
|:---|:-------|:-----|:-------|:---------|
| load_data | `uk_load_node_time` | UNIQUE | node_id, time | 同一拓扑节点同一时间点唯一 |
| load_data | `idx_load_node_time` | INDEX | node_id, time | 按节点和时间查询负荷 |
| load_data | `idx_hour` | INDEX | hour | `GROUP BY hour` 聚合 |
| load_data | `idx_day_of_week` | INDEX | day_of_week | 周末 vs 工作日对比 |
| load_data | `idx_month` | INDEX | month | 月度统计 |
| prediction_result | `idx_predict_time` | INDEX | predict_time | 查询未来 24h 预测 |
| prediction_result | `idx_created_at` | INDEX | created_at | 取最新一批预测 |
| alert_event | `idx_trigger_time` | INDEX | trigger_time | 告警时间线 |
| alert_event | `idx_level` | INDEX | level | 按级别筛选 |
| alert_event | `idx_is_read` | INDEX | is_read | 未读告警计数 |
| alert_rule | `idx_active` | INDEX | is_active | 查询启用规则 |
| model_version | `idx_active` | INDEX | is_active | 查询活跃模型 |
| grid_node | `idx_grid_node_parent` | INDEX | parent_id | 查询上下级节点 |
| grid_node | `idx_grid_node_type` | INDEX | node_type | 按节点类型筛选 |
| grid_edge | `uk_grid_edge` | UNIQUE | from_node_id, to_node_id, topology_version | 防止重复拓扑连接 |
| user | `idx_username` | UNIQUE | username | 登录查询 |

### 索引设计原则

- **节点时间唯一**：多节点数据允许同一时间存在多条记录，但同一 `node_id + time` 组合必须唯一
- **覆盖查询避免回表**：高频统计查询（hour/day/month 聚合）建立冗余字段 + 索引
- **低基数列不建索引**：`is_read` 只有 0/1，但单独索引在 `WHERE is_read=0` 时有效
- **不超过 5 个索引 / 表**：避免写入性能下降

## 6. 模拟数据说明

### 6.1 生成策略

```python
# generate_mock_data.py 核心逻辑
# 日负荷曲线 = 双峰形态 + 随机噪声 + 周末偏移 + 季节性趋势

for day in range(730):  # 2 年
    for hour in range(24):
        base = get_hourly_pattern(hour)      # 双峰：早 10:00 + 晚 18:00
        noise = random.gauss(0, 30)           # 高斯噪声 σ=30MW
        weekend_discount = 0.85 if is_weekend(day) else 1.0
        seasonal = 1.0 + 0.15 * sin(2π * day / 365)  # 夏高冬低
        
        load = (base * weekend_discount * seasonal) + noise
```

## P0-05 告警运行态与送达指标

`alert_runtime_state` 是可恢复的当前运行状态，业务键为规范化的 `state_key = nodeId:ruleId:type`，并有唯一约束。它保存 `PENDING`、`ACTIVE`、`SUPPRESSED`、`RECOVERED` 等生命周期、连续超限起点、冷却截止时间、活跃事件和最近已消费的来源序列。`alert_event` 是不可变的 occurrence 历史事实；恢复后再次触发会创建新的 occurrence，并通过 `previous_alert_id` 关联前一条事件。

V18 为 `alert_event` 追加 `state_key`、`occurrence_no`、规则/证据快照、实时来源与时间戳、推送时间、拓扑模拟标识和恢复/复发关联字段。历史记录允许为 `NULL`，读取时按 `LEGACY_EVIDENCE_MISSING` 处理，绝不批量伪造证据。规则和证据快照在触发时写入，不由规则更新接口回写。

`alert_delivery_metric` 使用 `(alert_id, user_id)` 唯一约束保存浏览器呈现确认。`latency_ms` 的口径是 `source_observed_at` 到服务端 `ack_received_at`；客户端时间仅用于诊断。合法样本按 nearest-rank 计算 P50/P95/max，缺少源时间或时钟倒退的记录排除在统计外。

## P0-04 实时遥测补充

Flyway `V17__realtime_telemetry.sql` 新增 `realtime_telemetry`，用于秒级模拟遥测审计和单实例重启后的最近 30 分钟恢复，不与小时级训练表 `load_data` 混用。字段包含 `node_id`、`observed_at`、`received_at`、`source_instance_id`、`sequence`、负荷与天气 DECIMAL 字段、质量、来源、估算、持久化状态和短原因。

唯一键 `(data_source, source_instance_id, sequence)` 负责幂等；`(node_id, observed_at)` 用于快照与恢复；`received_at` 用于每小时有界清理超过 24 小时的数据。当前来源仅为 `MOCK_REALTIME`；未接入 SCADA、Kafka 或多实例一致性。

## P0-06 模型产物与发布元数据

Flyway `V19__model_artifact_release.sql` 为 `model_version` 增加 `artifact_dir`、`artifact_checksum`、`runtime_status`、`deployed_at`、`last_load_error`、`last_health_checked_at`。状态包含 `ACTIVE`、`CANDIDATE`、`LEGACY_UNVERIFIED`、`LOAD_FAILED`、`ROLLBACK_REQUIRED`。`artifact_dir` 只允许是配置模型根目录下的单一版本目录名；旧记录字段为空时保持 `LEGACY_UNVERIFIED`，不伪造 checksum。当前方案是单 Flask 进程的原子 bundle reload，不是蓝绿部署或多实例高可用。

## P0-08 集成验收同步

当前数据库迁移版本为 `V1`～`V20`，不得重编号或修改已执行迁移。`model_version` 中 `Prophet` 记录只作为历史兼容数据保留；新的训练、发布和回滚入口仅适用于 LSTM。模型发布前必须同时满足不可变目录、manifest 文件清单、逐文件 SHA-256、artifact checksum、Flask 冒烟推理和健康版本一致性，数据库发布失败时旧运行模型继续服务。

预测批次 `forecast_run` 与 `prediction_result.forecast_run_id` 一起形成追溯边界；`RUNNING`、`FAILED`、`LEGACY` 记录不进入固定提前量复盘。实时遥测、告警运行态、送达确认和工单反馈均保留历史兼容缺失标记，不用默认值伪造样本或准确率。
