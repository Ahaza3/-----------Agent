# 🤖 AI 代码审查 SOP — 标准操作流程

> **版本**：v1.0 ｜ **日期**：2026-07-13 ｜ **负责人**：全员  
> **适用阶段**：Sprint 1 – Sprint 3（Day 4 – Day 15）  
> **关联考核**：AI 代码审查能力（权重 **30%**）

---

## 一、为什么 AI Code Review 占 30%？

本届实训的核心命题不再是"能不能写出功能"，而是：

> **"AI 写出代码之后，你能不能看懂、审查、优化它？"**

| 能力层级 | 要求 | 考核方式 |
|:---------|:-----|:---------|
| L1 识别 | 能发现 AI 代码中的明显 Bug（NPE、越界、类型错误） | Bug 注入评测 |
| L2 分析 | 能判断代码架构是否合理、是否存在安全隐患 | AI 审查 + 人工确认记录 |
| L3 优化 | 能提出改进方案并落地（性能、可读性、可维护性） | PR Review 评论质量 |
| L4 掌控 | 禁飞区代码逐行可讲，理解每一行的设计意图 | 答辩日抽查 |

## 二、整体流程

```
┌──────────────┐    ┌──────────────────┐    ┌──────────────────┐    ┌──────────┐
│  Developer   │    │   AI Reviewer    │    │  Human Reviewer  │    │  Merge   │
│  提交 PR     │───▶│  自动审查        │───▶│  人工确认         │───▶│  合并    │
│              │    │  (Claude Code)   │    │  (另一成员)       │    │          │
└──────────────┘    └──────────────────┘    └──────────────────┘    └──────────┘
                           │                        │
                    输出：Review 报告          确认 / 驳回 / 要求修改
                    标记：🚨/⚠️/💡             记录在 PR 评论中
```

### 2.1 流程步骤

| 步骤 | 操作者 | 动作 | 产出 |
|:-----|:-------|:-----|:-----|
| ① 提交 PR | Developer | `git push` → GitHub PR，填写 PR 描述模板 | PR 链接 |
| ② AI 审查 | Claude Code | 运行 `/code-review`，覆盖正确性 + 安全 + 简化 | Review 报告 |
| ③ 人工确认 | Reviewer（另一成员） | 逐条确认 AI 发现的问题，补充遗漏项 | PR 评论 |
| ④ 修复 | Developer | 修改代码，回复评论，标注 `fixed` | 新 commit |
| ⑤ 复审 | Reviewer | 确认修复，Approve → Merge | 合并到 develop |

### 2.2 时间要求

| 类型 | AI 审查 | 人工确认 | 总计（目标） |
|:-----|:-------:|:-------:|:------------:|
| 小改动（<200 行） | < 1 min | 5–10 min | < 15 min |
| 中改动（200–800 行） | 1–3 min | 10–20 min | < 25 min |
| 大改动（800+ 行） | 3–5 min | 20–40 min | < 45 min |

## 三、AI 审查工具与命令

### 3.1 Claude Code（主工具）

本项目使用 Claude Code 作为 AI 审查引擎。在项目根目录打开终端即可使用：

```bash
# 基础审查（中等强度，推荐日常使用）
claude "/code-review"

# 高强度审查（Bug 注入评测前 / Sprint 评审前）
claude "/code-review effort=high"

# 最深度审查（Sprint 结束 / 禁飞区复查）
claude "/code-review effort=max"

# 审查 + 自动修复
claude "/code-review --fix"
```

### 3.2 审查维度

每次审查自动覆盖以下维度：

| 维度 | 权重 | 关注点 |
|:-----|:----:|:-------|
| 🐛 **正确性** | 40% | NPE、边界条件、并发安全、类型错误、逻辑错误 |
| 🔒 **安全性** | 25% | SQL 注入、XSS、敏感信息泄露、权限校验缺失 |
| 📐 **架构规范** | 20% | Controller→Service→Mapper 分层、DTO 使用、MyBatis-Plus 规范 |
| ♻️ **简化度** | 15% | 冗余代码、可简化的逻辑、无用的抽象 |

### 3.3 辅助工具

```bash
# 单独检查简化度（不找 Bug，只做代码质量优化）
claude "/simplify"

# 安全专项审查
claude "/security-review"
```

## 四、审查检查清单（Checklist）

### 4.1 Java / Spring Boot（后端）

#### 🐛 正确性
- [ ] `@Transactional` 是否加在需要事务的方法上？只读方法是否标记 `readOnly = true`？
- [ ] Controller 参数是否有 `@Valid` / `@NotNull` 校验？
- [ ] Mapper 自定义 SQL 是否有 SQL 注入风险（应使用 `#{}` 而非 `${}`）？
- [ ] Stream / Optional 操作是否处理了空值（`orElseThrow` vs `orElse(null)`）？
- [ ] 日期时间是否统一使用 `LocalDateTime`（禁止 `java.util.Date`）？
- [ ] 缓存 key 是否有过期时间和命名空间隔离？

#### 🔒 安全性
- [ ] 接口是否有 JWT 鉴权（非公开 API 不得遗漏 `@PreAuthorize`）？
- [ ] 异常处理是否不会泄露内部堆栈信息给前端？
- [ ] 敏感配置（API Key、DB 密码）是否从 `.env` 读取，未硬编码？

#### 📐 架构规范
- [ ] 代码是否放在正确的包路径下（`controller/ → service/impl/ → mapper/`）？
- [ ] Controller 是否只做参数校验和路由，逻辑在 Service？
- [ ] 是否使用 `R<T>` 统一响应包装？
- [ ] 是否使用了 DTO 而非直接暴露 Entity？
- [ ] SQL 日志是否使用 `DEBUG` 级别（生产环境禁止打印完整 SQL）？

#### ♻️ 简化度
- [ ] 是否存在只调用一次就被封装的私有方法（过度抽象）？
- [ ] Lambda 是否可以替换为方法引用（`User::getName` 替代 `u -> u.getName()`）？
- [ ] 是否有未使用的 import / 变量 / 方法？

---

### 4.2 React / TypeScript（前端）

#### 🐛 正确性
- [ ] `useEffect` 是否有正确的依赖数组（无遗漏、无多余）？
- [ ] 异步请求是否有 loading / error 状态管理？
- [ ] 组件卸载后是否取消了未完成的请求（`AbortController` 或 cleanup）？
- [ ] ECharts 实例是否在组件卸载时 `dispose()`？
- [ ] TypeScript 是否有 `any` 逃逸（特殊情况需注释说明）？

#### 🔒 安全性
- [ ] 用户输入的搜索/查询参数是否做了编码（防 XSS）？
- [ ] Token 是否存储在 `httpOnly` cookie 或安全内存中（非 `localStorage`）？

#### 📐 架构规范
- [ ] 状态管理是否使用 Zustand（非 Redux），store 是否按领域拆分？
- [ ] API 调用是否通过统一的 `request.ts` 封装（含错误拦截）？
- [ ] 组件是否遵循 Ant Design 的 `props` 命名惯例？
- [ ] 图表配置是否抽取为独立的 `useXxxChartOption` hook？

#### ♻️ 简化度
- [ ] 是否有可抽取为自定义 hook 的重复逻辑？
- [ ] JSX 中是否有超过 30 行的复杂条件渲染（应提取子组件）？
- [ ] 是否存在可合并的重复 `useState` 或 `useEffect`？

---

### 4.3 Python / ML（机器学习）

#### 🐛 正确性
- [ ] 数据加载是否有空 DataFrame 保护？
- [ ] 训练/推理时的特征列顺序是否一致？
- [ ] `train_test_split` 或时间序列切分逻辑是否正确（时间序列禁止随机切分）？
- [ ] 模型输入 shape 是否与训练时匹配？

#### 📐 架构规范
- [ ] Flask 推理服务是否提供了 `/health` 端点供 Java 侧健康检查？
- [ ] 模型加载是否在 Flask 启动时完成（不在每次请求时加载）？
- [ ] 推理结果是否包含置信度或误差带？

## 五、🚫 AI 禁飞区特殊审查规则

3 个禁飞区模块 **AI 只能审查，不能生成修复代码**：

| 编号 | 模块 | 文件 | AI 审查方式 |
|:-----|:-----|:-----|:------------|
| 🚫 NFZ-1 | Agent 核心循环 | `AgentCore.java` | AI 可标记问题位置 + 问题类型，**不给出修复代码**；开发者自行修复后，Reviewer 逐行确认 |
| 🚫 NFZ-2 | Prompt 工程 | `buildSystemPrompt()` | AI 可审查 Prompt 结构合理性，**不给出修改后的 Prompt 文本** |
| 🚫 NFZ-3 | 告警检测引擎 | `ThresholdDetector.java` + `AlertTemplate.java` | AI 可检查逻辑漏洞，**不给出修复代码** |

### 禁飞区审查流程（不同于普通代码）

```
AI 标记问题（仅描述，不给代码）
    │
    ▼
开发者自行修复（手写修改）
    │
    ▼
Reviewer 逐行检查（确保开发者真正理解）
    │
    ▼
答辩日导师抽查（逐行讲解 → 无法讲解 = 模块零分）
```

### 禁飞区 PR 标签

禁飞区代码的 PR 标题必须加 `[NFZ]` 前缀，Reviewer 看到此标签即进入禁飞区审查流程。

## 六、审查结果分级与处理

### 6.1 问题严重级别

| 标记 | 含义 | 处理要求 | 示例 |
|:-----|:-----|:---------|:-----|
| 🔴 **Block** | 必须修复才能合并 | 立即修复 | NPE、SQL 注入、数据丢失风险 |
| 🟡 **Warn** | 强烈建议修复 | 修复或提供不修理由 | 性能隐患、不规范但有原因 |
| 🔵 **Info** | 建议性改进 | 可讨论，不阻塞合并 | 命名优化、注释补充 |

### 6.2 合并准入标准

| 审查项 | 最低标准 |
|:-------|:---------|
| 🔴 Block 问题 | **0 个** |
| 🟡 Warn 问题 | ≤ 3 个（超过需 Reviewer 特批） |
| 禁飞区代码 | 必须走禁飞区审查流程（见第五章） |
| 人工确认 | 至少 1 名 Reviewer Approve |
| CI 通过 | GitHub Actions build + test 全部通过 |

## 八、Bug 注入评测演练流程

每个 Sprint 评审日进行 Bug 注入评测（30 分钟限时），流程如下：

```
                    ┌─────────────┐
导师注入 3 个 Bug    │  30 分钟限时  │
（Sprint 评审前 1h）─▶│  团队定位修复  │──▶ 导师评分
                    └─────────────┘
                          │
                    ┌──────┴──────┐
                    │  AI 辅助定位 │  ← 允许使用 Claude Code 审查
                    │  人工决策修复 │  ← 不可无脑接受 AI 修复方案
                    └─────────────┘
```

### 演练准备（每个 Sprint 结束前 1 天）

```bash
# 1. 对整个 Sprint 的改动做一次深度审查
claude "/code-review effort=max"

# 2. 查看审查报告中的 🔴 问题，人工确认后修复
# 3. 团队集体过一遍发现的 Bug 模式，积累经验
```

### 常见注入 Bug 模式（5 类必考）

| 类型 | 示例 | 出现概率 |
|:-----|:-----|:--------:|
| NPE / 空值处理 | 未判空的 Optional.get() 或对象引用 | ⭐⭐⭐⭐⭐ |
| 边界条件 | 数组越界、时间范围开闭区间错误 | ⭐⭐⭐⭐ |
| 并发安全 | 未加锁的共享状态、线程不安全集合 | ⭐⭐⭐ |
| 资源泄漏 | 未关闭的流、未 dispose 的图表实例 | ⭐⭐⭐ |
| SQL / 注入 | `${}` 代替 `#{}`、缺少参数校验 | ⭐⭐ |

## 九、团队角色与职责

| 角色 | Code Review 职责 |
|:-----|:-----------------|
| 🔧 **技术经理（Dev）** | 后端代码主 Reviewer；禁飞区代码最终确认人；Bug 注入评测主力 |
| 🎨 **产品经理（PO）** | 前端代码主 Reviewer；UI/UX 一致性和交互体验审查 |
| 🧪 **测试工程师（QA）** | 测试代码审查；API 契约一致性校验；CI/CD 流水线维护 |
| 🎯 **项目经理（SM）** | 审查流程监督；PR 模板合规检查；Sprint 评审日审查质量统计 |

### 轮值规则

- 每个 PR 至少 1 名 Reviewer（不能是 PR 作者本人）
- 每日 Review 负载均衡：每人每天最多 Review 2 个中大型 PR
- 禁飞区代码必须由 **技术经理** 做最终确认

## 十、Git 工作流集成

### 10.1 分支策略

```
main ────────────────────────────────────────────────
  │
  └── develop ──────────────────────────────────────
        │         │
        │         └── feature/xxx ──▶ PR ──▶ (AI Review) ──▶ (Human Review) ──▶ merge
        │
        └── feature/yyy ──▶ PR ──▶ (AI Review) ──▶ (Human Review) ──▶ merge
```

### 10.2 日常操作指令

```bash
# 1. 从 develop 创建功能分支
git checkout develop && git pull && git checkout -b feature/your-feature

# 2. 开发完成后 push
git add . && git commit -m "feat: 描述你的改动"
git push -u origin feature/your-feature

# 3. 在 GitHub 创建 PR（使用模板）
# → CI 自动运行 build + test
# → 运行 AI 审查: claude "/code-review"
# → 指定 Reviewer

# 4. Reviewer 审查通过后合并
# → Squash & Merge 到 develop

# 5. 删除功能分支
git branch -d feature/your-feature
git push origin --delete feature/your-feature
```

### 10.3 Commit 规范（Conventional Commits）

```
feat: 新功能
fix:  Bug 修复
docs: 文档变更
refactor: 重构（不改变功能）
test: 测试相关
chore: 构建/工具/依赖变更
style: 格式化（不影响逻辑）
perf: 性能优化
```

## 十二、常见问题 FAQ

### Q1: AI 审查结果和人工判断冲突怎么办？
**A:** 人工判断优先。AI 是辅助工具，不是最终决策者。若 AI 误报，在 PR 评论中说明理由并标记 `wontfix`。

### Q2: 小改动是否可以跳过 AI 审查？
**A:** 不可以。所有 PR 必须经过 AI 审查。小改动的 AI 审查通常 < 1 分钟，不构成时间负担。

### Q3: 禁飞区代码是否能给 AI 看？
**A:** 可以。AI **可以看**、可以审查、可以指出问题。但 AI **不能给出修复代码**——修复必须由开发者手写完成。

### Q4: CI 未通过的 PR 能否进入 Review？
**A:** 不能。PR 作者应确保 CI 全部通过后再请求 Review。

### Q5: 紧急 Hotfix 如何走流程？
**A:** Hotfix 分支从 `main` 切出，同样需要 AI 审查 + 人工确认，但可申请加速（Reviewer 优先响应）。合并时需要同时合入 `main` 和 `develop`。
